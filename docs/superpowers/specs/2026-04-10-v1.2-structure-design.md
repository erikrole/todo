# v1.2 Structure Design

## Overview

Three related features shipped together: **sub-projects** (projects nested one level deep), **sections** (named groups within a project), and **section reordering** (drag sections around). Together they let a project grow internal structure without becoming unwieldy.

---

## Data Model

### `projects` table — new column

```sql
parent_project_id TEXT REFERENCES projects(id)
```

Nullable. A project with a `parent_project_id` is a sub-project. The 1-level depth limit is enforced at the API layer (a project whose parent already has a parent is rejected). No cascade delete — deleting a parent project also deletes its sub-projects (same behavior as deleting any project today).

### `sections` table — new table

| Column | Type | Notes |
|---|---|---|
| `id` | TEXT PK | nanoid |
| `project_id` | TEXT NOT NULL FK → projects | |
| `title` | TEXT NOT NULL | |
| `position` | REAL NOT NULL | fractional index |
| `is_collapsed` | INTEGER NOT NULL DEFAULT 0 | 0 = expanded, 1 = collapsed |

One row per section. Sections belong to exactly one project. Sub-projects can have sections; parent projects can have sections.

### `tasks` table — new column

```sql
section_id TEXT REFERENCES sections(id)
```

Nullable. Tasks with `section_id = NULL` are unsectioned. Sections exist only within projects — tasks not in a project cannot have a `section_id`.

**On section delete:** `section_id` is set to NULL on all tasks in that section (unsectioned, not deleted).

**On project delete:** Tasks are deleted (existing behavior). Sections are deleted via cascade.

## Shared Types

```typescript
// New
interface Section {
  id: string;
  projectId: string;
  title: string;
  position: number;
  isCollapsed: boolean;
}

// Extended
interface Project {
  // existing fields...
  parentProjectId: string | null;  // new
}

interface Task {
  // existing fields...
  sectionId: string | null;  // new
}
```

Schemas: `CreateProjectSchema` gains optional `parentProjectId`; `CreateSectionSchema` and `UpdateSectionSchema` are new; `UpdateTaskSchema` gains nullable `sectionId`.

## API Layer

### Projects (extend existing)

**`GET /api/projects`** — response shape gains `parentProjectId`.

**`POST /api/projects`** — accepts optional `parentProjectId`. Validation: the referenced parent must exist and must itself have `parentProjectId = NULL` (1-level limit).

### Sections (new)

**`GET /api/sections?projectId=X`** — returns sections for a project ordered by `position` ascending.

**`POST /api/sections`** — body: `{ projectId, title, position }`. Returns created section.

**`PATCH /api/sections/:id`** — partial update: `title`, `position`, `isCollapsed`. Used for rename, reorder, and collapse toggle.

**`DELETE /api/sections/:id`** — NULLs `section_id` on all tasks in the section, then deletes the section.

### Tasks (extend existing)

**`PATCH /api/tasks/:id`** — gains `sectionId` as a nullable field. Setting `sectionId: null` moves the task to unsectioned.

## TanStack Query Hooks

Four new hooks mirroring the sections API:

- `useSections(projectId)` — `GET /api/sections?projectId=X`
- `useCreateSection()` — `POST /api/sections`
- `useUpdateSection()` — `PATCH /api/sections/:id`
- `useDeleteSection()` — `DELETE /api/sections/:id`

Existing `useUpdateTask` already handles partial updates — no new hook needed for moving tasks between sections.

## Sidebar Tree

### Data

`useProjects()` returns all projects flat. The sidebar groups them client-side:

```
top-level projects = projects where parentProjectId === null
sub-projects = grouped under their parent by parentProjectId
```

No extra fetch.

### Rendering

Each top-level project with sub-projects renders:

```
[chevron] [color dot] Project Name
  [color dot] Sub-project A
  [color dot] Sub-project B
```

- Clicking the chevron toggles sub-project visibility (collapse state stored in `localStorage`, keyed by project id, expanded by default).
- Clicking the project name navigates to the project.
- Sub-projects are indented; they have no chevron (no grandchildren).
- Top-level projects without sub-projects render as today (no chevron).

### Active State

When viewing a sub-project: the sub-project item shows the full active style; its parent shows a muted highlight.

### Right-click Menu

Existing project context menu gains: **"Add Sub-project"** — opens an inline input scoped to that parent. Hidden on sub-project items (1-level limit).

## Project Page — Sections

### Layout

```
[unsectioned tasks — no header]
[New Task quick-add]

── Section A ────────────────── ▾  (2)  ⋮
  [task]
  [task]
  [New Task quick-add]

── Section B ────────────────── ▾  (1)  ⋮
  [task]
  [New Task quick-add]

[+ Add Section]
```

Unsectioned tasks sit at the top with no header. Each section renders a header row followed by its tasks and a per-section quick-add. At the bottom: an "Add Section" button.

### Section Header Row

Left to right: drag handle (⠿), section title, chevron (▾/▸), task count badge, right-click menu trigger (⋮).

### Creating a Section

Two entry points — both produce the same inline text input at the bottom of the section list:
1. Click "Add Section" button.
2. Right-click anywhere on the project background (not on a task) → context menu → "Add Section".

Type a name → Enter to save, Escape to cancel. New section gets `position = max(existing) + 1`.

### Collapsing a Section

Click the chevron on the section header. Collapsed state is persisted via `PATCH /api/sections/:id` (server-side, not localStorage — sections are server entities). Collapsed sections show only the header row.

### Reordering Sections

Drag the drag handle on a section header using `@dnd-kit` (same library as task drag-and-drop). On drop, recalculate fractional position (midpoint between the two neighbors) and fire `PATCH /api/sections/:id` with the new position. Tasks within the section move with it — their own positions are unchanged.

### Moving Tasks Between Sections

The existing task right-click context menu gains a new item: **"Move to Section"** → submenu listing all sections in the project + "No Section" at the top. Only shown when the task is in a project that has at least one section. Fires `PATCH /api/tasks/:id` with the target `sectionId` (or `null` for unsectioned). No drag-and-drop between sections in v1.2.

### Section Right-click Menu

- **Rename** — title becomes an inline text input; Enter to save, Escape to cancel.
- **Delete** — if the section has tasks, confirm: *"This will unsection [N] tasks. Continue?"* On confirm, calls `DELETE /api/sections/:id`.

---

## Out of Scope (v1.2)

- Drag-and-drop tasks between sections
- Sections in areas, Inbox, Today, or Upcoming views
- Sub-project nesting beyond 1 level
- Section templates
- Collapsing sub-projects in the parent project page (sub-projects are sidebar-only)
